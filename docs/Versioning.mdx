# Versioning

Each myui component is versioned independently. When you install a component you get a snapshot of its source — it lives in your repo and you own it. Versioning answers two questions:

- **Which npm packages should be installed?** — dep ranges are pinned in the registry so `shadcn add` always installs a tested combination.
- **What changed since I last installed this?** — every component ships a `CHANGELOG.md` you can read to manually apply upstream changes to your local copy.

---

## Checking a component's version

The changelog for each component is shown directly on its Storybook page — look for the collapsed **Changelog** section at the bottom of the docs tab. It includes a link to the raw file on GitHub.

Every item in `registry.json` also carries a `version` field you can inspect directly:

```bash
curl https://idos350.github.io/myui/r/button.json | jq '.meta.version'
```

---

## Updating a component

Because you own the source, there is no automatic update — but there are two paths depending on whether you have modified the component.

### Unmodified component

If you have not changed the component since installing it, `shadcn update` does a clean replacement:

```bash
pnpm dlx shadcn@latest add @myui/button --diff      # preview the diff first
pnpm dlx shadcn@latest add @myui/button --overwrite  # apply it
```

### Modified component

If you have made local changes, start with the diff — it is still the fastest way to spot structural changes:

```bash
pnpm dlx shadcn@latest add @myui/button --diff
```

Then use the changelog to understand _why_ each change was made and whether it applies to your copy:

1. **Find when you last installed the component** — you can use `git log --oneline -- src/components/ui/<your-component>` to find the relevant commit.
2. Open the component's **Changelog** section on its Storybook docs page, or follow the GitHub link there.
3. Read every entry dated _after_ your install commit — those are the changes you haven't received.
4. Apply the relevant ones to your local copy by hand, using the diff as a reference.

Each changelog entry includes the exact file and location so you can apply it in under a minute:

```markdown
## [1.2.0] - 2026-06-15

### Fixed

- Focus ring no longer bleeds outside border-radius on Safari
  — added `overflow: hidden` to `.root` in `Button.module.scss`
```

## npm dependency pinning

When `shadcn add` runs, it installs the npm packages listed in the registry item's `dependencies` array. myui pins these to `^` ranges sourced from the version of the package that the registry was built and tested against:

```json
"dependencies": ["@base-ui/react@^1.5.0", "clsx@^2.1.1"]
```

This means your package manager will install a compatible minor/patch version rather than resolving whatever is latest at install time. If you later upgrade a dep independently, the components will continue to work as long as the major version is the same.

---

## Semver conventions

Versions are per-component — bumping `Button` to `2.0.0` has no implication for `Accordion`.

| Change                                                         | Bump          |
| -------------------------------------------------------------- | ------------- |
| Bug fix, style tweak, no API change                            | patch `1.0.1` |
| New prop, new variant, backward-compatible addition            | minor `1.1.0` |
| Removed/renamed prop, restructured files, changed import shape | major `2.0.0` |
